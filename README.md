# Task Service

Сервис для управления задачами с HTTP API на Go.

## Требования

- Go `1.23+`
- Docker и Docker Compose

## Быстрый запуск через Docker Compose

```bash
docker compose up --build
```

После запуска сервис будет доступен по адресу `http://localhost:8080`.

Если `postgres` уже запускался ранее со старой схемой, пересоздай volume:

```bash
docker compose down -v
docker compose up --build
```

Причина в том, что SQL-файлы из `migrations/` монтируются в `docker-entrypoint-initdb.d` и применяются только при инициализации пустого data volume.

## API

Базовый префикс API:

```text
/api/v1
```

Маршруты:

- `POST   /api/v1/tasks`
- `GET    /api/v1/tasks`
- `GET    /api/v1/tasks/{id}`
- `PUT    /api/v1/tasks/{id}`
- `DELETE /api/v1/tasks/{id}`
- `GET    /api/v1/tasks/{id}/occurrences` — даты выполнения задачи

## Функциональность периодичности задач

### Концепция

Задачи могут быть разовыми или периодическими. Настройки периодичности хранятся прямо в задаче (поле `recurrence`). Копии задач не создаются — даты вычисляются на лету по запросу.

### Поля задачи

| Поле | Описание |
|---|---|
| `scheduled_at` | Срок выполнения разовой задачи или якорная дата для типа `daily` |
| `recurrence` | Опциональные настройки повторяемости (null = задача без повторов) |

### Типы повторяемости

#### `daily` — каждые N дней

Задача повторяется каждые `interval` дней, начиная с `scheduled_at`.  
Поля: `interval` (int, >= 1). Требует `scheduled_at`.

```json
{
  "title": "Обзвон пациентов",
  "scheduled_at": "2026-04-01T09:00:00Z",
  "recurrence": { "type": "daily", "interval": 1 }
}
```

#### `monthly` — каждое N-е число месяца

Задача выполняется в указанный день каждого месяца.  
Поля: `day_of_month` (int, 1–30). Месяцы, в которых нет такого числа (например, февраль для 30-го), пропускаются.

```json
{
  "title": "Формирование отчётности",
  "recurrence": { "type": "monthly", "day_of_month": 5 }
}
```

#### `dates` — конкретные даты

Задача выполняется только в перечисленные даты.  
Поля: `dates` (array of "YYYY-MM-DD", непустой).

```json
{
  "title": "Инвентаризация",
  "recurrence": { "type": "dates", "dates": ["2026-04-15", "2026-07-15", "2026-10-15"] }
}
```

#### `even_odd` — чётные/нечётные числа месяца

Задача выполняется во все чётные или все нечётные числа месяца.  
Поля: `parity` ("even" | "odd").

```json
{
  "title": "Обход пациентов",
  "recurrence": { "type": "even_odd", "parity": "odd" }
}
```

### Получение дат выполнения

```
GET /api/v1/tasks/{id}/occurrences?from=2026-04-01&to=2026-04-30
```

Query-параметры `from` и `to` принимают даты в формате `YYYY-MM-DD`.  
По умолчанию: `from` = сегодня, `to` = сегодня + 30 дней.  
Максимальный диапазон — 1 год, максимум 366 дат в ответе.

Пример ответа:

```json
{
  "task_id": 1,
  "from": "2026-04-01",
  "to": "2026-04-07",
  "occurrences": ["2026-04-01", "2026-04-03", "2026-04-05", "2026-04-07"]
}
```

## Тесты

```bash
go test ./internal/usecase/task/...
```

Покрытие: все 4 типа повторяемости, граничные случаи (февраль без 30-го числа, пустой диапазон, диапазон до/после якоря, невалидные форматы дат, невалидные параметры).

### Интеграционные тесты (HTTP + PostgreSQL)

Поднимают временный Postgres в Docker через [Testcontainers](https://golang.testcontainers.org/), применяют те же SQL из `migrations/`, стартуют HTTP-роутер и проверяют ключевые сценарии API.

Требуется **запущенный Docker**. Перед первым запуском на чистой машине выполните `go mod tidy`, чтобы подтянуть зависимости и пополнить `go.sum`.

```bash
go mod tidy
go test -tags=integration -count=1 ./integration/...
```

Обычный `go test ./...` интеграционные тесты не включает (build-тег `integration`).

## Принятые решения и допущения

1. **Нет автоматического создания копий задач.** Задача создаётся один раз. Endpoint `/occurrences` вычисляет даты по правилу. Это проще, не засоряет БД, и позволяет менять правило без каскадных обновлений.

2. **`day_of_month` ограничен 1–30, не 1–31.** 31-е число есть только в 7 из 12 месяцев, что делает его малополезным для ежемесячных задач. Ограничение 30 гарантирует минимум 11 выполнений в год.

3. **Для типа `monthly` при отсутствии числа в месяце — пропуск, не перенос.** Например, задача на 30-е число: февраль пропускается. Перенос на ближайший день создал бы неожиданное поведение.

4. **`scheduled_at` обязателен только для `daily`.** Для остальных типов он опционален и служит как срок выполнения (отображается в задаче, не влияет на расчёт дат).

5. **Хранение `recurrence` как JSONB.** Разные типы повторяемости имеют разные поля. JSONB гибко хранит их без множества NULL-колонок в реляционной схеме. Значение корректно маршализуется/демаршализуется через `encoding/json`.

6. **Нет миграционного runner'а.** Проект использует монтирование `.sql` файлов в `docker-entrypoint-initdb.d`. При добавлении новой миграции нужно пересоздать volume: `docker compose down -v && docker compose up --build`.
